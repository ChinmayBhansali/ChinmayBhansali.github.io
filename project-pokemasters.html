Phase 0

Project Planning

The Notebook Origins

PokeMasters started as a physical notebook game I created during grade 9 while attending boarding school. Electronic devices were strictly prohibited in the hostel, prompting this offline, paper based RPG. I played it endlessly with my dorm mates.

Years later, as a university software development project, I decided to digitize this childhood creation into a fully fledged Java application. The goal was to build a single player RPG where players could explore the world, catch wild characters, engage in turn based combat, and save their game state to return to later.

Core Objective

Translate pen and paper combat mechanics into structured, object oriented code.

Player Journey

Define user stories: start new games, choose starters, battle, catch, heal, and quit.

Phase 1

Domain Model & Console UI

Establishing the functional baseline

Before any graphics were introduced, the core logic needed to be bulletproof. This phase focused on building the domain model: designing the foundational Java classes like Player, Pokemon, and Attack. I utilized extensive inheritance hierarchies to manage the different characters and abilities.

To test these mechanics, I developed a functional text based Command Line Interface (CLI). Through the console, players could navigate menus, engage in battles, calculate health point reductions based on attack stats, and execute catching logic using PokeBalls.

The functional text-based CLI inside the IntelliJ console (click to enlarge).

Phase 2

Data Persistence

Saving and Loading the Game State

An essential feature of any RPG is the ability to save progress and return to it later. I implemented JSON based data persistence to achieve this. By designing a Writable interface along with JsonReader and JsonWriter classes, the application could systematically serialize the entire GameModel.

This allowed players to save their exact inventory, current party of active fighters, stored characters, and financial balance to a local JSON file, ensuring seamless continuation across play sessions.

Phase 3

Graphical User Interface

Transitioning to a Visual Desktop Experience

This phase marked a major architectural shift. I transitioned the project from the terminal to a visual desktop application using Java Swing. To cleanly separate the internal logic from the user interface, I implemented the Model View Controller (MVC) pattern.

I also utilized the Observer Pattern to handle event driven updates. Whenever the underlying BattleModel changed (like taking damage or catching a wild creature), it automatically notified the GameGraphics UI to update health bars, battle logs, and sprites in real time without convoluted dependencies.

The final Java Swing GUI featuring health bars, battle logs, and action menus (click to enlarge).

Phase 4

Event Logging & Design Reflection

Refactoring a massive Object-Oriented hierarchy

The final phase focused on implementing a robust event system using the Singleton Pattern. An EventLog tracked all critical in-game actions chronologically (for example, printing "Pikachu was caught!" with a precise timestamp to the console upon exit).

More importantly, this phase required a critical reflection on the software architecture. As visible in the UML diagram, the project suffered from poor cohesion in certain areas. Creating a distinct rigid sub-class for every individual creature and attack type resulted in class explosion. My primary takeaway for future refactoring was to utilize databases or HashSets to store character stats, drastically improving flexibility over rigid inheritance.
